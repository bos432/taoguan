# admin-next 与 uni-app 联动上线计划（含前端实施、协议勾选验收、数据隔离）

## 概要
本次上线不只是后台 `admin-next`，还要把 `zflUniApp` 一并纳入前端上线范围，按“**可正式运营、但不影响现在线上数据**”为硬约束推进。

当前已确认：

- 后台源码基座：`zflAdminWeb`
- 商家端源码：`zflMerchantWeb`
- C 端前端：`zflUniApp\zflUniApp`
- `zflUniApp` 是 **uni-app 多端前端**，覆盖 H5、微信小程序，并兼容 APP 打包
- 登录协议勾选逻辑当前已满足：**默认不勾选，未勾选禁止登录**

本轮上线建议按“双前端并行验收”执行：

- 后台：`admin-next`
- C 端：`zflUniApp`
- 数据环境：**独立测试库**
- 发布策略：**新目录灰度 / 新入口灰度**
- 目标周期：**20 到 28 个工作日**

## 前端实施范围
### 一、后台前端 `admin-next`
周期：**12 到 16 个工作日**

- 继续以 `zflAdminWeb` 为可维护源码基座。
- 完成后台页面从“统一壳子”到“可运营页面”的补齐。
- 优先完成商品、订单、会员、商家、系统、权限等核心运营模块。
- 继续逐栏目完成内容、文件、溯源、巡检、分析、导出等页面承接。
- 保留旧入口承接和路由兼容，避免影响现有后台操作路径。
- 构建产物继续区分 `local / dev / online`，正式上线前先走灰度目录。

**重点前端交付**
- 菜单与路由稳定
- 页面无死链、无白屏
- 批量操作、审核、禁用、迁移、回显、异常提示完整
- 页面交互习惯尽量贴近当前线上 `admin-next`

### 二、uni-app 前端 `zflUniApp`
周期：**6 到 8 个工作日**

- 核查 H5 与小程序端登录、协议、首页、商品、购物车、结算、订单、我的、商家申请等核心页面。
- 统一登录前的协议确认逻辑，确保所有登录入口行为一致。
- 检查协议中心、用户协议、隐私政策链接跳转是否正常。
- 检查 H5 与小程序环境下登录流程是否一致，尤其是：
  - 手机验证码登录
  - 微信授权登录
  - 账号密码登录
- 对结算页、商家申请页等含协议勾选的页面一并回归，确保不出现默认已勾选。
- 如需调整，仅允许做前端显式行为收口，不改坏现有用户登录链路。

**已确认验收项**
- 登录页协议默认未勾选
- 未勾选协议不可登录
- 协议中心、用户协议、隐私政策可打开
- 登录成功后才进行协议接受记录补提交流程

## 数据不受影响的隔离方案
### 联调与测试
- 后台前端和 uni-app 前端统一接入**独立测试库 / 测试后端**。
- 所有新增、编辑、审核、迁移、结算、申请类写操作，只能在测试环境验证。
- 若需核对正式环境，只允许只读检查，不允许带写操作联调。
- 任何正式环境验收都不得默认把写接口直接暴露给开发中的前端包。

### 正式发布
- 后台 `admin-next` 先发到新目录灰度，再切正式目录。
- H5 端也先发灰度入口，再切正式 `/app/`。
- 小程序端先在测试版/体验版验证，通过后再提交正式版。
- 必须保留旧后台和旧 H5 入口的可回退能力。

### 接口约束
- 不改线上数据库结构，除非单独走后端变更评审。
- 不替换已有线上接口语义，只允许兼容性新增。
- 所有前端环境变量与接口地址明确分离，禁止测试包误连正式可写库。
